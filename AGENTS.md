# Project Instructions (RFC - Local C++ Framework)

You are working on a C++ codebase that exceeds your context limit.
To preserve context memory, follow these strict execution rules.

## 1. Do Not Blindly Read Source Files
Never use the `read` tool on raw `.cpp` or `.h` / `.hpp` files unless you have
already identified the specific file and reason to read it (implementing,
debugging, or modifying an implementation). Do not open files "just to check."

## 2. Consult the Map First
A structural skeleton of the framework's namespaces, classes, inheritance,
and other declarations is available in `./symbols.md`. Always read `symbols.md`
before reading any source file.

`symbols.md` is a **navigation index**, not documentation. It is generated by a
deterministic extraction pass (ctags/regex) plus a local LLM summarization
pass, so treat it as follows:

- **Class, Function, Struct, Macro, Union, Enum names and file paths are reliable** — trust
  these to locate code.
- **`symbols.md` does not capture implementation details, full doc comments, or
  private logic.** It only covers declarations visible in headers.

## 3. Your Workflow

1. Read `symbols.md` first to locate the relevant namespace, class, function, 
   struct, macro, union and enum names for the task at hand.
2. Use the `File:` path listed for the class to identify exactly which file
   to open next. Do not guess file paths from class names — `symbols.md` is the
   canonical source for file locations in this codebase, since folder
   structure does not always match naming conventions.
3. If a class or symbol you need does not appear in `symbols.md`, it may be
   missing from the index (e.g. newly added, or inside a macro-heavy region
   ctags/regex failed to parse). In that case, ask the user or fall back to
   a targeted search (e.g. grep) rather than reading large files wholesale.

## 4. Updating the Map

If you modify, add, or remove a Class, Function, Struct, Macro, Union, Enum in a header file
within rfc folder, note this at the end of your response so `symbols.md` can be regenerated for
the affected file(s). Do not attempt to hand-edit `symbols.md` yourself.

## 5. Coding Style

This framework does not use c++ exceptions, therefore no std containers will be used within the code.
All functions are marked as noexcept. Use the same coding style as in header/source files.
KString was optimized to use with unicode strings. So, use unicode strings instead of ansi.
KString does not support for multiple zero terminated strings.
wchar_t buff[32];
KString txt(buff); should not be used. it will treat as literal.
instead, use this: KString txt(buff, KStringBehaviour::MAKE_A_COPY); 
or this macro: STR_COPY(buff);
see `rfc/core/KString.h` file for more info.
